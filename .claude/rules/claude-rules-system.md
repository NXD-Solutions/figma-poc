# Claude Rules System

## Where rules live

Global rules (applying to all NXD repos) belong in `.claude-static/rules` in the `.github` repo. Updates to static rules are made via PR to `.github`.

## Local testing

`~/.claude/rules/` may be used for local testing before submitting a PR, but `.claude-static/rules` is the source of truth. The rules owner decides what enters `.claude-static/rules` — Claude does not self-promote rules.

## Triggers

Before committing any new or changed rules file, check the filename suffix:
- `.local.md` — repo-specific, never promote. Skip the trigger entirely.
- `.gen.md` — autogenerated, never edit or promote manually.
- `.md` — ask: does this rule apply to all NXD repos? If yes, notify the user and propose a PR to `.claude-static/rules` before proceeding.

## Proposing new rules

When Claude identifies a missing or needed rule, it notifies the local user: "I have identified a Claude rules gap: `<finding>`. Shall we request a change?" On approval, Claude opens a PR to `.github`. The PR description must cover:
- **What** the rule is
- **Why** it is needed
- **Value** it adds
- **Risks** any concerns or potential side effects

## Promoting rules to .claude-static/rules

On user approval, follow these steps exactly:

### 1. Prepare the branch
```
# Get main SHA
gh api repos/NXD-Solutions/.github/git/refs/heads/main | python3 -c "import sys,json; d=json.load(sys.stdin); print(d['object']['sha'])"

# Create feature branch
gh api repos/NXD-Solutions/.github/git/refs -X POST \
  -f ref="refs/heads/feature/promote-<name>" \
  -f sha="<main-sha>"
```

### 2. For each file to promote
```
# Check if file already exists (get SHA if so, 404 if not)
gh api "repos/NXD-Solutions/.github/contents/.claude-static/rules/<file>.md?ref=feature/promote-<name>" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('sha','NEW'))"

# Base64-encode the file
cat .claude/rules/<file>.md | base64 -w 0

# Create (no sha field) or update (sha field required)
gh api repos/NXD-Solutions/.github/contents/.claude-static/rules/<file>.md \
  -X PUT --input - <<EOF
{"message":"<Add|Update> <file>","content":"<base64>","branch":"feature/promote-<name>"}
EOF
# For update, add: ,"sha":"<existing-sha>"
```

### 3. Create the PR
```
gh pr create \
  --repo NXD-Solutions/.github \
  --base main \
  --head feature/promote-<name> \
  --title "<concise title>" \
  --body "..."
```

PR body must cover **What**, **Why**, **Value**, **Risks** — reason from the actual file content, not generic placeholders. Blast radius assessment must identify which repos are affected and whether any may not match the rule's assumptions.

### Notes
- If `claude-rules-system.md` itself was modified locally, include it in the same PR
- `jq` is not available — use `python3` to parse JSON responses
- Output the PR URL when done

## Blast radius

Changes to `.claude-static/rules/` are distributed to all repos in the NXD-Solutions org. Review with extra care before merging.

## Commands

Command files live in `.claude/commands/`. When writing the `description` frontmatter field, end the description with `(NXD)` — this makes the command discoverable when typing `/nxd` in the Claude Code command picker.

## Autogenerated files

Files with the `.gen.md` suffix are autogenerated — never edit them manually.

## Local-only files

Files with the `.local.md` suffix are repo-specific and must never be promoted to `.claude-static/rules`. The trigger does not fire for these files.
